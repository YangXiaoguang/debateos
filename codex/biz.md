# 产品定位与基本描述
一、这个产品的核心定位
你要做的，其实不是“让几个 Agent 一起说话”，而是：
让用户预设的多个 Agent，围绕同一个议题进行结构化、多轮、可追踪、可裁决的讨论，最终产出胜出观点、胜出 Agent，以及完整的推理过程。
它适合的场景非常多，比如：
战略方案讨论
产品需求评审
技术路线争论
投资项目评估
科研方案比较
政策文本分析
风险识别与反驳验证
所以这个系统的价值，不在“热闹”，而在三个关键词：
对抗性推理、结构化收敛、可解释胜出。
二、产品的基本工作流设想
1. 用户先定义 Agent 阵容
用户不是临时拉几个机器人进群，而是配置一组“可复用角色”。
每个 Agent 至少要有这些属性：
名称
身份设定
立场/偏好
专业领域
推理风格
发言约束
可调用知识源/工具
胜负倾向参数
例如：
保守派架构师 Agent
激进创新派产品官 Agent
风险控制官 Agent
成本核算专家 Agent
用户体验批判者 Agent
裁判 Agent / 主持人 Agent
这样一来，系统就不是一次性问答，而是一个 Agent 编排平台。
2. 用户提出议题
议题不是普通问题，而是“需要争论、比较、交锋、收敛”的题目。
比如：
是否应该采用私有化大模型替代现有规则引擎？
住建行业公共收益治理平台应该先做资金透明还是流程监督？
某科研路线 A 和路线 B，哪条更值得投入？
系统可以要求用户补充：
目标
评价标准
时间约束
成本约束
禁止条件
胜出规则
这一步很重要，因为它决定讨论不是发散闲聊，而是围绕“裁决目标”推进。
3. 进入多轮讨论
真正的核心在这里。
一个高质量的系统，不应该只是“轮流说一遍”，而应该有清晰的讨论协议，比如：
第一轮：立场陈述
每个 Agent 先给出自己的基本观点、核心论据、初步方案。
第二轮：相互质询
每个 Agent 可以点名质疑其他 Agent 的逻辑漏洞、成本忽略、风险遗漏、前提错误。
第三轮：防守与修正
被攻击的 Agent 必须回应，允许修改自己的主张，而不是永远嘴硬。
第四轮：交叉比较
系统自动把各方观点整理成对照表，让 Agent 针对关键差异继续交锋。
第五轮：收敛陈词
每个 Agent 用简洁方式总结：
最终主张
为什么自己更优
为什么对手不够好
这时，系统已经不是“多 Agent 聊天”，而是 议题驱动的辩论状态机。
4. 决出胜出者
“胜出”必须机制化，否则最后只是热闹而没有价值。
可以有几种裁决方式：
方式A：裁判 Agent 判定
由专门的 Judge Agent 根据预设规则打分，例如：
逻辑完整性
证据质量
风险意识
可执行性
成本合理性
与用户目标一致度
方式B：用户最终投票
适合创意、策略、文案、品牌等主观性更强的议题。
方式C：混合裁决
系统分数 + 用户权重 + 裁判意见，共同决定胜者。
方式D：淘汰赛机制
如果 Agent 比较多，可以采用：
初赛分组讨论
半决赛交锋
决赛一对一终局辩论
这个机制会非常有产品感，也容易形成差异化。
三、这个系统最重要的产品特色，不是“多”，而是“可控”
如果只是多个模型轮流发言，很快会失控、重复、空话很多。
所以必须做“讨论协议层”。
我建议系统至少有这几个核心控制层：
1. 主持人层
负责：
控制轮次
分配发言权
限制跑题
强制回到议题
标记冲突点
推动进入下一阶段
也就是 Debate Orchestrator。
2. 记忆与状态层
系统必须记录：
当前轮次
各 Agent 主要立场
已提出的关键证据
已暴露的争议点
未解决的问题
每个 Agent 的信誉分/表现分
否则讨论到后面会越来越乱，前后矛盾也无法处理。
3. 评分与裁判层
这是你产品的灵魂之一。
系统不能只输出“谁赢了”，还要输出：
为什么赢
赢在什么维度
输在哪里
哪个论点最关键
哪一轮决定了胜负
这样才会有真正的“可解释裁决”。
4. 结果沉淀层
每一次讨论结束后，不只是生成聊天记录，而是沉淀为结构化资产：
议题名称
参与 Agent
主要论点树
关键冲突点
最终胜者
胜出原因
失败观点总结
可复用模板
这会让产品从“会话工具”升级为“组织认知资产系统”。
四、建议你第一阶段先做成什么样
不要一开始就做特别复杂的开放式博弈系统。
第一版最好做成一个 可控的 MVP。
MVP 建议包含 5 个核心页面
1. Agent 配置页
创建/编辑 Agent：
名称
角色说明
系统指令
立场标签
发言风格
工具权限
2. 议题创建页
输入：
议题
目标
胜出规则
参与 Agent
最大轮次
是否允许引用知识库
3. 讨论现场页
以时间线方式展示：
当前轮次
谁在发言
谁在反驳谁
当前争议焦点
当前积分
4. 裁决结果页
展示：
胜出 Agent
最终结论
关键论点图谱
分数明细
裁判解释
5. 讨论归档页
可以按议题、Agent、领域检索历史讨论。
五、从系统设计角度，建议你定义三种产品模式
1. 自由讨论模式
更开放，更像圆桌讨论。
适合头脑风暴、创意探索。
2. 辩论对抗模式
强制正反或多方立场对抗。
适合方案比较、决策评审。
3. 锦标赛模式
多个 Agent 多轮淘汰，最后产生冠军。
这个模式最有“产品辨识度”和演示效果。

# 产品prd描述
一、PRD v1
1.1 产品名称
暂定名：多 AI Agent 议题辩论与裁决系统
英文工作名：DebateOS / Agent Debate Arena
1.2 产品定位
一个允许用户自定义多个 AI Agent，围绕指定议题进行多轮结构化讨论、批判、回应、裁决，并最终输出胜者、胜出内容与过程资产的系统。
它不是普通聊天产品，而是：
议题驱动
协议驱动
状态机驱动
裁决驱动
结果资产化驱动
1.3 产品目标
核心目标
帮助用户把复杂议题从“单次问答”升级为“多视角对抗式求解”，提升结论质量、透明度与可复盘性。
MVP 目标
先完成一版可稳定运行的受控讨论系统，支持：
创建和管理自定义 Agent
创建议题并上传文本/图片/文件
选择 2–4 个 Agent 参与讨论
支持开始、暂停、恢复、中断
支持多轮流式输出
支持裁判判定与胜者生成
支持复制、收藏、归档
商业/产品价值
提升复杂问题分析深度
降低单一模型偏见
沉淀组织讨论资产
为后续 Agent 平台化提供基础
1.4 用户角色
角色一：普通使用者
负责创建议题、选择 Agent、查看结果。
角色二：高级配置者
负责创建/配置 Agent、维护 Prompt、设定模型与风格。
角色三：管理员
负责系统配置、模型权限、全局参数、内容审核、用量管理。
1.5 典型使用场景
场景 A：技术方案评审
用户把“是否采用某技术路线”作为议题，选择架构师 Agent、成本 Agent、风控 Agent、创新 Agent 进行讨论。
场景 B：产品需求辩论
围绕某产品方向，让保守派、增长派、用户体验派进行攻防。
场景 C：品牌/文案/创意竞争
多个创意 Agent 互相批判并给出更优版本，最终选出最佳方案。
场景 D：研究思路比较
多个研究型 Agent 从不同理论与方法路径对同一问题展开讨论。
1.6 产品边界
MVP 内
单个议题会话
2–4 个 Agent
固定讨论协议
1 个裁判机制
文件作为辅助材料输入
单轮流式输出
按轮次暂停与恢复
历史会话归档
MVP 外
多裁判并行
自动学习型 Agent
跨会话 Agent 自我演化
自动联网检索
复杂企业权限分层
团队协作审批流
大规模淘汰赛锦标赛
1.7 核心业务规则
1.7.1 议题创建规则
用户创建议题时，可输入：
标题
议题描述
附加文本
附件
图片
输出要求
胜出规则
最大轮次
1.7.2 Agent 规则
每个 Agent 由用户提前定义，至少包含：
名称
头像
模型
系统指令
风格说明
立场标签
能力限制
1.7.3 参与规则
最少 2 个 Agent 才能开始
支持手动选择
支持随机选择
随机选择时用户输入数量，系统从候选池中抽取
数量不能小于 2
MVP 建议最大 4 个
1.7.4 讨论流程规则
默认采用四阶段协议：
开场立场陈述
批判对手观点
回应与修正
最终陈词与裁决
1.7.5 认输规则
Agent 可以主动认输
认输后状态为 conceded
认输后不再参与后续轮次
认输是行为事件，不是唯一胜负条件
1.7.6 胜出规则
MVP 默认采用 混合模式：
若只剩一个活跃 Agent，则直接胜出
若到达轮次上限仍有多个活跃 Agent，则由裁判评分决出胜者
胜者输出标记为“胜出内容”
1.7.7 中断/暂停规则
暂停：保留当前会话与轮次状态，可继续
恢复：从最近一个可恢复检查点继续
中断：终止当前会话，不继续后续生成
中断后可发起重新讨论，但不覆盖历史记录
