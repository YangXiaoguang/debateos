# 这版 UI 的核心问题

不是“某几个控件没画好”，而是 **信息架构、首页职责、主任务路径、视觉层级** 四件事同时挤在一个页面里，导致用户一进来就被迫处理太多事情。

从高级产品经理和 UI/UX 设计师的角度看，这个页面目前更像一个“内部演示台”或“设计稿拼贴页”，还不像一个可高效完成任务的产品首页。

---

# 一、先给总判断：这版首页承担了太多职责

当前页面同时塞进了这几类内容：

1. 品牌/导航
2. 全局搜索
3. 最近辩论列表
4. 首页大标题与说明
5. 创建 Agent 表单
6. 创建 Topic 表单
7. 系统健康监控
8. 执行说明
9. 管理入口

这会带来一个非常典型的问题：

**用户不知道当前页到底是“首页 / 工作台 / 新建页 / 监控页 / 帮助页 / 管理页”的哪一种。**

而一个优秀的 B 端或专业工具型产品，首页必须回答三个问题：

* 我现在在哪
* 我现在最该做什么
* 做完第一步后下一步去哪

这张图里的首页，对这三个问题都回答得不够清楚。

# 二、具体问题拆解

## 1. 左侧导航的问题最明显：层级混乱、按钮拥挤、品牌与导航混在一起

你红框标出来的区域，确实是当前页面最先暴露问题的地方。

### 具体问题

第一，最顶部那个 “DO” 更像品牌 Logo，但视觉上又像一级菜单按钮。
用户会疑惑：它是返回首页、品牌入口、工作台，还是一个功能按钮？

第二，`ARENA / AGENTS / TOPICS / ARCHIV` 全部做成同一种胶囊按钮，优先级几乎一样。
这会导致用户看不出：

* 哪个是当前页面
* 哪个是主导航
* 哪个是次级入口

第三，左侧导航区太窄，文字被压缩得很重，阅读效率低。
尤其是英文全大写配合高字距，在窄栏里更难读。

第四，导航和内容之间缺少明显的“选中态”和“位置反馈”。
用户很难一眼知道当前在哪个模块。

第五，品牌、主导航、次导航、底部用户入口的分层不够清晰。
这会让左侧栏看起来像一串竖着堆的按钮，而不是系统导航。

### 本质问题

**导航不是在“组织页面”，而是在“堆按钮”。**

---

## 2. 首页的职责严重过载：应该简洁，但现在像控制台 + 表单页 + 文档页的混合体

首页中间区域同时放了：

* 一段很大的宣言式文案
* 创建 Agent 表单
* 创建 Topic 表单
* 多个状态标签
* 输入框、下拉框、数字框、上传入口

这会导致两个问题。

### 问题一：用户刚进首页就被迫填写复杂表单

这是典型的“让用户一上来就工作”，而不是“先帮用户建立清晰路径”。

首页更适合做：

* 概览
* 最近任务
* 快捷入口
* 主 CTA

而不是直接承载完整创建动作。

### 问题二：创建 Agent 和创建 Topic 同时出现，打乱认知顺序

虽然文案写了“先配置角色，再定义议题”，但界面上两个大表单并排出现，用户还是会下意识同时看两边，注意力被切碎。

### 本质问题

**首页没有聚焦一个主任务，而是在同时推销多个任务。**

---

## 3. 信息层级不清：视觉上“看起来都很重要”，结果就是“没有真正重要的东西”

当前页面里有很多视觉元素都在抢主视觉：

* 中间大标题区很强
* 左侧 Recent Debates 列表很长
* 右侧 System Health 四个卡片也很强
* 右侧 Execution Notes 又是大块信息
* 下方两个表单占据大量视觉面积

所以会发生一个经典问题：

**页面上处处都是重点，用户反而找不到重点。**

一个高质量专业产品界面，通常会严格控制：

* 一级视觉焦点：1 个
* 二级功能组：2–3 个
* 三级信息：按需展开

但这版页面几乎所有模块都在争一级注意力。

---

## 4. 交互路径不自然：用户的任务流没有被界面“引导”

当前产品的真实主流程应该是：

1. 创建或选择 Agent
2. 创建或选择 Topic
3. 组装 Debate
4. 开始辩论
5. 进入实时会话
6. 查看结果与归档

但这个页面没有把这条路径真正设计出来。

### 现在的问题

* “创建 Agent”和“定义 Topic”并排展示，但没有流程关系
* 没有明显的“下一步”动作
* 没有进度感
* 没有告诉用户当前还差什么
* 没有突出“开始辩论”这个真正的核心动作

### 本质问题

**界面在展示功能，但没有在组织流程。**

---

## 5. 右侧信息区过重：System Health、Execution Notes、Admin Entry 不应该在首页这么突出

从产品优先级上看：

* System Health 是运营/管理信息
* Execution Notes 更像内部指导文档
* Admin Entry 是管理入口

这些都不该和“创建 Agent / Topic”争夺首页主位置。

这类内容应当是：

* 可折叠
* 次级可见
* 进入特定角色后可见
* 或者移动到专门页面

尤其是 `Execution Notes`，它本质上像“使用说明”或“团队内部备注”，直接放在首页会非常像后台原型，而不是成熟产品。

---

## 6. Recent Debates 列表过长，占据了大量首屏注意力

左侧中栏的 Recent Debates 区域现在非常像一个消息流，但内容重复度高，且信息密度不均衡。

### 问题点

* 三个卡片内容几乎雷同
* 首屏重复信息太多
* 占据高度大，但支持决策的信息少
* 没有强过滤，没有状态分组，没有快速操作

### 更深层问题

最近辩论是重要模块，但不应该在首页以“长列表主角”的形式出现。
更合理的是：

* 只展示 3 条最关键的
* 提供 “Continue / View All” 明确动作
* 把历史会话交给独立页面

---

## 7. 表单设计偏“堆字段”，不够像专业工作流产品

当前 Step 1 / Step 2 的表单有几个典型问题。

### 第一，用户输入负担太大

一上来就要求填写：

* 名称
* 角色说明
* Prompt
* 模型
* 风格
* 立场
* 背景
* 议题描述
* 约束条件
* 输出要求
* 轮次

这对第一次使用者负担太重。

### 第二，字段的主次关系不清

比如：

* “Agent 名称”与“大段系统指令”同时占很大面积
* “轮次数字”过于裸露
* 高级设置没有被收纳到折叠层级里

### 第三，缺少模板与默认值

对于这种专业系统，应该尽量减少用户从零开始配置的压力。

---

## 8. 文本可读性与视觉舒适度还有优化空间

当前视觉风格整体是对的：
深色、科技、专业、带控制台气质。

但问题在于：

### 对比度问题

很多次级文字偏灰，字重又轻，在深色背景下可读性一般，尤其长时间使用会累。

### 全大写英文标签过多

如 `STEP 1 / STRUCTURED / RECENT DEBATES / SYSTEM HEALTH`。
这种风格适合作为装饰标签，但不能太多，否则会让页面显得“样式感很强，效率感不足”。

### 边框和容器过多

几乎每块都包在一个发光边框/深色卡片里，导致页面很“满”。
视觉上的呼吸感不够。

# 三、从产品角度，真正应该怎么重构

我建议把整个产品的 IA 重新整理成下面这套。

## 一级导航只保留 5 个

### 1. Home / 工作台

只做概览与继续工作，不做完整创建。

### 2. Agents

管理 Agent 资产。

### 3. Topics

管理议题与附件。

### 4. Debates

管理辩论会话、进行中会话、历史结果。

### 5. Archive / Reports

查看归档、结论、导出结果。

管理后台、模型配置、系统健康，统一放右上角设置或 Admin 区，不进入主导航核心层。

# 四、优秀的首页应该长什么样

## 首页只保留 4 个核心区块

### 区块 1：主欢迎区 + 主 CTA

只做三件事：

* 显示一句清晰的产品价值主张
* 展示当前用户最近状态
* 提供一个最主要按钮

例如：

**开始一场新辩论**
次按钮：

* 从模板创建
* 继续上次会话

### 区块 2：快速工作流入口

用 3 个步骤卡，而不是完整表单：

* 配置 Agent
* 定义 Topic
* 开始 Debate

每个步骤只显示：

* 当前数量
* 最近更新时间
* 进入按钮

而不是把完整表单都摊开。

### 区块 3：最近会话

只展示 3 条，且每条给明确动作：

* 继续
* 查看结果
* 复制配置

### 区块 4：系统状态摘要

只保留轻量版统计：

* Agents 数
* Topics 数
* Running Sessions
* Models Ready

不要在首页展开 Execution Notes 和大块管理信息。

# 五、优秀的创建流程应该怎么设计

## 不要在首页并排放两个大表单

建议改成 **引导式三步流**。

### 方案 A：向导式 Wizard

用户点击“开始新辩论”后，进入 3 步流程：

#### Step 1：选择或创建 Agent

* 左侧 Agent 列表
* 右侧详情预览
* 支持快速新建
* 支持模板复制

#### Step 2：选择或创建 Topic

* 标题
* 描述
* 附件上传
* 高级设置折叠

#### Step 3：组装并确认

* 选择 2–4 个 Agent
* 确认轮次
* 确认模式
* 点击开始

### 方案 B：右侧抽屉式 Quick Create

首页只保留按钮。
点击后右侧弹出抽屉完成轻量创建。
适合效率型产品。

我更推荐方案 A，因为你的系统不是简单发起会话，而是有明确结构和准备过程。

# 一、首页重新定义：它到底负责什么

先把首页职责定死，否则后面再怎么改都会继续膨胀。

## 首页的唯一职责

首页只负责三件事：

**第一，告诉用户系统当前状态。**
我有哪些 Agent、有哪些 Topic、最近有哪些 Debate、系统是否可用。

**第二，帮助用户快速进入主流程。**
开始一场新的 Debate，或者继续一场已有 Debate。

**第三，提供最少但最关键的导航分发。**
把用户引导到 Agents、Topics、Debates、Archive，而不是在首页完成所有事情。

---

## 首页明确不做什么

这些内容不应该作为首页主内容出现：

* 完整 Agent 创建表单
* 完整 Topic 创建表单
* 大段执行说明
* 大块 Admin 入口
* 重度系统健康监控
* 长列表历史归档

这些都应该去子页面，或者以抽屉 / 向导 / 二级页承载。

# 二、优化后的首页信息架构

我建议首页采用 **三栏结构保留，但彻底重构职责**。

## 1. 左侧：全局导航栏

左侧只承担“全局导航”，不承担内容展示。

### 导航结构

从上到下：

**顶部品牌区**

* Logo

# DebateOS UI Audit

## 当前问题

1. 页面之间缺少共享骨架。登录页、工作台、辩论页、模型页各自维护自己的容器、间距和视觉层级，导致体验像多个独立工具拼在一起。
2. 同类信息没有统一组件。状态、卡片、表单、空态、指标、步骤标题反复用不同写法实现，造成视觉噪音和维护成本。
3. 工作流优先级不够清楚。首页既承担资源总览又承担创建流程，但“先做什么、后做什么”没有被足够强调。
4. 高信息密度页面缺少分层。辩论工作区把回合内容、裁判结果、附件证据、事件轨迹同时摆在一个平面里，阅读成本偏高。
5. 移动端阅读顺序不理想。原先在窄屏下侧栏和导航很容易抢在核心任务前面，弱化了主流程。

## 推荐解法

1. 建立共享 UI 基座，统一 page canvas、frame shell、surface card、status badge、action button、input、metric tile。
2. 把页面分成三类区域：导航轨、上下文侧栏、主工作区、洞察侧栏，并让 Dashboard 与 Debate Workspace 共用同一布局逻辑。
3. 把所有复杂表单统一成“SectionTitle + FieldLabel + Input + InlineHint”的结构，让信息顺序固定下来。
4. 为实时界面建立明确层级：会话控制与指标在顶部，turn 时间线放中间，裁判与产物沉淀放右侧，议题与附件证据放左侧。
5. 以移动端优先顺序重新排布页面，让主任务先出现，再展示导航与辅助信息。

## 本次落地原则

- 视觉方向：深色控制台、青蓝高亮、强对比卡片层。
- 字体策略：展示层用 `Space Grotesk`，指标与系统标签用 `IBM Plex Mono`。
- 组件原则：优先复用 `console-kit`，避免再次出现大量散落的单页专用样式。
- 信息原则：每张卡片只服务一个认知目标，不再混放不同优先级内容。
- 反馈原则：所有状态变化都尽量用统一 badge、按钮 disabled 态和空态说明表达。

## 后续约束

- 新页面优先从共享 shell 和 primitives 组合，不要重新手写整页容器。
- 新增表单必须补 label 与 hint，不能只放 placeholder。
- 新增状态展示优先扩展统一 badge tone，而不是临时写一套颜色。
- 大于两列的信息页必须先定义主次区域，再决定卡片布局。
